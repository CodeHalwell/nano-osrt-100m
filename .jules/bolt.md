
## 2024-05-30 - Repetition penalty vectorization in generation loop
**Learning:** PyTorch models generating text autoregressively can be heavily bottlenecked if they iterate over generated tokens using python loops (`set(generated[0].tolist())`) on every step to apply penalties (e.g. repetition penalty). This not only scales linearly with context length (O(N) operation per generated token, totaling O(N^2) for the full generation), but also causes a severe performance drop. In `hf_model.py`, the old approach took over 1 second for a sequence of 4000 tokens on batch size 1! Also, hardcoding `generated[0]` causes the penalty to only be applied correctly when the batch size is 1.
**Action:** Vectorize penalty logic using `mask.scatter_` to boolean mask the `next_logits` directly, ensuring you handle out-of-bounds indices by safely allocating mask memory up to `max(vocab_size, max_gen_id)` before slicing back to `vocab_size` because `generated` can contain token indices outside `next_logits`'s vocab size (e.g., padding). Also wrap operations in `torch.where` to handle logic safely without loops.
